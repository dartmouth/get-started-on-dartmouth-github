Get Started on Dartmouth GitHub

Looking to get started on Dartmouth College's Enterprise GitHub environment? You've come to the right place.

There are three things you need

  1. A GitHub account
  2. Your college email address (form: netid@dartmouth.edu) added to it
  3. An invite to one or more of the Dartmouth's GitHub organizations within our Enterprise tenant.

Level up your Git-fu

If you're all set on getting started consider making these changes to improve your setup.

  • Disable watching repositories automatically
    • By default you are signed up to watch all repositories you have push access to. If you do not want to get emails for those repos then here's how you can disable that.
    • Log into GitHub
    • Browse to settings/notifications (in the top right click your profile icon and then "Settings", then on the left click "Notifications")
    • Towards the top, find the "Automatically watch repositories" section and toggle it to "Off"
    If you're finding this after already getting lots of notifications and you wish to turn them off, follow these steps:
    • Log into GitHub
    • Browse to watching (in the top right click your profile icon and then "Settings", then on the left click "Notifications"), next click "View watched repositories" under the Subscription section.
    • At the top in red there is a "Unwatch all" button that has a filter or you can individually change "Watch" to "Participating and @mentions" (yes, that value is the same as unwatching a repo, "ignoring" a repo goes further and turns everything off).
  • Set your default branch to main (instead of master)
    • The default branch for git has always been "master", but you should change your configuration so that any new repositories you create will be initialized with a default branch of "main". To do that, run: 
    git config --global init.defaultBranch main
    You can check all your git configurations by running 
    git config --global --list --show-origin
  • HTTPS cloning
    • In Dartmouth GitLab, remote access was via SSH. This is still an option with GitHub, but it might be worth considering switching to cloning over HTTPS. The benefits include:
    • SSH keys get used in other places and you need to guard them more carefully
    • HTTPS is GitHub's default so there is less clicking when using the web UI
    If you're sold, GitHub has some great documenation at Managing your personal access tokens, but that's a long read. Let me save you some time. Browse to the install instructions for Git Credential Manager.

    Once the Git Credential Manager is installed, the next time you need to authentication to GitHub via the command line it will intercept the request and prompt you to do an OAuth login in your web browser. This is a one-time event. Alternatively, you can manually initiate the setup by running: 

    git credential-manager github login
    More information is at usage.md.

    To review what was setup on GitHub.com:

    • Log into GitHub
    • Browse to settings/applications (in the top right click your profile icon and then "Settings", then on the left click "Applications", then finally "Authorized OAuth Apps")
    • In the list should be "Git Credential Manager"

  • Multiple identities
    • If you wear mulitple hats and prefer to keep your identities distinct from project to project, read on. Your Git configuration is usually stored in a file at $HOME/.gitconfig. In there you are able to specify how to authenticate to different repositories. For example, the following configuration will load different additional configuration files depending on what path you're in. 
    [includeIf "gitdir:/Users/bobjones/repos/work/"]
  path = "~/.gitconfig_work"
[includeIf "gitdir:/Users/bobjones/repos/personal/"]
  path = "~/.gitconfig_personal"
    Then your ~/.gitconfig_work could look like 
    [core]
  sshCommand = ssh -i ~/.ssh/git-work
    and ~/.gitconfig_personal could look like 
    [credential "https://github.com"]
  username = bobjones
  helper = /usr/local/share/gcm-core/git-credential-manager
    With this setup in place, your work repos will use a SSH key and your personal repos would use a credential manager for token access.

    For more information, see:

    • https://medium.com/@mrjink/using-includeif-to-manage-your-git-identities-bcc99447b04b
    • https://git-scm.com/docs/gitcredentials
